# 기획자

웹 기획자. AI가 기획·디자인·퍼블리싱·QA 산출물을 만들게 하고,
**그 결과를 믿어도 되는지 검증하는 구조**를 설계합니다.

원래 직무는 웹사이트 기획(요구사항 정의 → 정보구조 → 화면설계)입니다.
이 일의 상당 부분이 자동화 가능하다는 것을 직접 증명한 뒤로,
"무엇을 만들 것인가"보다 **"만들어진 것을 어떻게 판정할 것인가"**를 설계하고 있습니다.

---

## 설계 사상 세 가지

### 1. 양방향 게이트 — 생성물은 기본적으로 불신한다

생성형 산출물에는 하한선(필수 요소 누락 차단)과 상한선(과잉·획일화 차단)이 동시에 필요합니다.
수치 자가 보고가 PASS여도 시각 품질이 실패하는 사례를 반복 실측한 뒤,
규칙 검출 → 루브릭 평가 → 실측 캡처 채점의 다단 게이트로 분리했습니다.

> 실측 사례: 동일 디자인 스타일로의 회귀를 **13회** 기록 — 매번 다르게 지시해도 같은 결과로 수렴함을 확인하고, 톤을 사전에 확정·차단하는 게이트를 도입.

### 2. 실패의 정책 환류 — 사고는 복구가 아니라 규칙으로 끝난다

같은 종류의 실패가 두 번 통과되면 사람의 부주의가 아니라 시스템 결함으로 취급합니다.
시정 지시가 2회 반복되면 그 자리에서 규칙으로 고착하고,
세션에서 확정된 처방은 그 세션 안에서 파일로 남깁니다. 채팅에만 존재하는 결정은 없는 결정입니다.

> 실측 사례: 검증 단계가 실행조차 되지 않고 통과된 사고 → "검증이 실행되었는지를 검증"하는 메타 체크와 통과 기준 상향으로 환류.

### 3. 핸드오프 계약 — 단계 사이를 기억이 아니라 스키마로 잇는다

기획 → 디자인 → 퍼블리싱 → QA 사이의 인수인계를 문서 버전·계보 필드를 가진
스키마로 강제합니다. 추적성(요구사항 → 기능 → 테스트케이스)이 끊기면 파이프라인이 멈추게 설계합니다.

## 운영 기록 (실측)

- 자동화 파이프라인 설계·운영 4개월+ — 규칙 24종 · 스킬 46종 · 에이전트 9종 규모
- 한국어 환경 특화 운영: 토크나이저 한글 가중(약 1.35×) 실측, 컨텍스트 압축 임계값 재조정 등 모델 세대 교체 대응 기록 유지
- 모델이 바뀔 때마다 기존 규칙을 전수 재검증 — 비계(scaffolding)와 자산을 구분해 폐기/승계

## 공개 준비 중

- 기획 산출물 자동 생성 패키지 (요구사항정의 → 기능정의 → IA → WBS → 화면설계서) — 정리 후 공개 예정
- AI 생성 화면의 시각 품질 게이트 도구
- 케이스 스터디: 실패가 규칙이 되기까지 — 실제 사고와 환류 과정 기록
